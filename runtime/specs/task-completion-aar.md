# Task Completion AAR

- A task is not complete until the main work is finished, verification is done, and the 30-second AAR scan has been answered.
- The AAR scan always asks four questions: new pattern, new trap, missing rule, and outdated rule. If all four answers are no, task completion may continue after the scan is recorded.
- If any answer is yes, task completion stays blocked until the lesson is recorded with a concise summary, a concrete detail note, and any required workflow follow-up.
- Lessons should only be promoted when at least 2 of these 3 checks pass: repeatable, expensive if unknown, and not already visible from code or official docs.
- Every recorded lesson must be generalized so a future agent can understand it outside the original project story. Convert the local discovery into a reusable pattern and state the consequence of ignoring it.
- High-cost lessons are not complete when merely stored in a reference file. They must also be activated on the workflow path, skill summary, rules file, or harness shell that a future task will naturally read.
- Treat these skip rationalizations as invalid: "this task is too small", "I will batch the AAR later", "the user is in a hurry", "I already know this lesson", and "the rule already exists so I do not need to scan".
- Stop immediately when any red flag appears: trying to skip AAR, declaring completion without the scan, writing gotchas without updating the completion workflow, or fixing the same class of bug twice without updating the rule set.
- Only narrow trivial cases may use a skip reason label: format-only, comment-only, dependency-version-only, or refactor-with-no-new-lesson. Even then, the scan itself is still part of task completion.
